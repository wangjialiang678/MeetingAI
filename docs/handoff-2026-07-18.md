## 项目是什么

Mac 原生 SwiftUI 桌面应用（SPM executableTarget，macOS 14+）：会议期间实时录音 → ASR 转写 → AI 自动分析产出洞察卡片，支持对话式交互。

当前真实技术栈（以此为准，部分旧文档过时）：

| 组件 | 当前事实 |
|---|---|
| 实时 ASR | 项目内 `asr-bridge/`（Go 子进程，端口 **18090**，由 `~/Library/Application Support/MeetingAI/config.json` 指定），DashScope `qwen3-asr-flash-realtime`，App 走 `ws://127.0.0.1:18090/v1/stream`。**18089 属于隔壁 SpeakLow，不要用**（2026-07-18 撞过一次，见 `docs/incident-asr-port-conflict-2026-07-18.md`） |
| AI 后端 | 默认 **HTTP**（2026-07-18 用户决策：Codex CLI 洞察 50s+ 太慢，降为可选）。本机 config.json 指定 `z-ai/glm-5.2` @ OpenRouter，key 用 `OPENROUTER_API_KEY`（`ai.apiKeyEnv` 可配）；代码默认 `qwen/qwen3.5-122b-a10b` @ NVIDIA |
| 说话人分离 | 双轨架构：实时轨不动；归档轨把录音按 chunk 封存单声道 WAV → OSS 上传 → Fun-ASR 非实时 HTTP → merge → 回填 `.diarized.jsonl` / `.transcript.md` / UI |
| UI | 左侧 `TranscriptView` 转写 + 右侧 `InsightFeedView` 洞察卡片流（不是旧的 ChatView） |
| 分析触发 | 按**文本长度**触发（partial 也计入），顾问模式 200 字 / 研究员模式 100 字；不再按 final 条数触发 |
| 密钥 | 全部从 `~/.claude/api-vault.env` 读取：`DASHSCOPE_API_KEY`（ASR）、`QWEN_API_KEY`（HTTP AI）。**缺 `OSS_ACCESS_KEY_ID` / `OSS_ACCESS_KEY_SECRET` / `MEETINGAI_DIARIZATION_UPLOAD_BUCKET`，所以 Fun-ASR 真实云端链路目前 BLOCKED** |

## 已知坑（改代码前先读）

- `pkill asr-bridge` 会误杀隔壁 SpeakLow 应用的同名进程，清理时按路径过滤（如 `pkill -f '会议中AI给建议.*asr-bridge'`）。
- 端口冲突（**2026-07-18 真实发作**：SpeakLow 占 18089，bind 通配竟成功、健康检查全绿、WS 无限重连）：复盘 + 排查 checklist 见 `docs/incident-asr-port-conflict-2026-07-18.md`，已用 config.json 把 MeetingAI 固定到 18090 规避。同日已落地三重代码加固：启动前 lsof 预检（`ASRBridgePortGuard`，自家残留清理/外来报错）、bridge 显式绑定 127.0.0.1（冲突立刻响亮失败）、`/health` 带 `service: meetingai-asr-bridge` 身份且启动流程校验（假绿快速失败）。
- `.txt` 只写 final；partial-only 长会要靠 `.transcript.md`。计划中的 `TranscriptStore` 尚未做。
- 系统消息已拆为独立 `.system` 卡片类型（2026-07-18）；上下文构建与 `.ai.md` 按 kind 区分，不再依赖 `[系统]` 前缀。
- NVIDIA/Qwen 的 OpenAI-compatible 响应正文可能在 `message.reasoning_content` 而非 `content`，解析已兼容，勿回退。
- Fun-ASR `speaker_id` 不保证跨 chunk 指同一真人；预签名 URL 一律不落日志。
