# 讨论: 从被动总结到主动研究助手

## 核心转变

当前 MeetingAI 是一个**定期总结器**——每隔一段时间把转写内容总结一遍。用户真正想要的是一个**主动参与的研究助手**：

| | 现在 | 期望 |
|---|---|---|
| AI 角色 | 被动总结 | 主动思考、主动提问、主动调研 |
| 输出节奏 | 定时/手动触发 | 有想法就说，有课题就去查 |
| 输出内容 | 总结已有内容 | 新洞察 + 新问题 + 调研结果 |
| 信息流向 | 会议→AI→总结 | 会议→AI→调研→新信息回流到会议 |

---

## 场景分析

### 场景 1: AI 发现值得深挖的点

> 会议中有人说"我们的转化率最近降了不少"。
>
> AI 注意到这个信号，在右侧面板轻轻冒出一条：
> **💡 转化率下降——是全行业趋势还是我们的问题？要不要我查一下最近行业的转化率基准数据？**
>
> 用户瞄一眼，点了"去查"。AI 开始后台调研。
> 会议继续。2 分钟后，AI 把调研结果推上来：
> **📊 调研完成：SaaS 行业 Q1 转化率普遍下滑 12-18%（来源: ...）。你们的情况可能不是个例。**

### 场景 2: AI 主动提出反直觉问题

> 会议讨论决定降价促销来提升销量。
>
> AI 插一句：
> **🤔 换个角度想——降价一定能提升总收入吗？如果你们的用户价格敏感度低，降价可能只是让利给了本来就会买的人。有没有考虑过提价 + 增加高端功能这条路？**
>
> 这个问题让讨论多了一个维度。

### 场景 3: 有人提到一个不确定的事实

> "好像 Apple 最近改了 App Store 的抽成比例？"
>
> AI 立即去查，几秒后：
> **🔍 快速核实：Apple 2024 年起对年收入 100 万美元以下的开发者抽成 15%，100 万以上仍为 30%。没有最近的新变更。**

### 场景 4: 讨论陷入僵局

> 团队在 A 方案和 B 方案之间争论了 10 分钟，没有进展。
>
> AI 观察到僵局模式：
> **⚡ 观察到 A/B 方案讨论了一会儿了。两个核心分歧是：1) 开发周期预期不同 2) 对用户接受度判断不同。要不要我分别调研一下这两个问题，拿数据说话？**

## 交互设计

### 核心理念：AI 是一个安静但敏锐的旁听者

会议场景的关键约束：**用户的注意力在会议上，不在 App 上**。所以 AI 的输出必须：

- 一眼能扫到重点（不需要细读）
- 不紧急的信息可以稍后看
- 需要决策时才打断（如"要不要去调研"）

### 右侧面板重新设计：信息流 + 研究队列

现有的聊天式面板不够用了。建议拆成两个区域：

```
┌─────────────────────────────────────────────────────────┐
│  顶部栏                                                  │
├────────────────────────┬────────────────────────────────┤
│                        │  ┌─ 信息流 ──────────────────┐ │
│                        │  │                            │ │
│   📝 实时转写           │  │ 💡 AI 洞察、提问、快速核实  │ │
│                        │  │    （主时间线，自动推送）    │ │
│                        │  │                            │ │
│                        │  ├─ 调研卡片 ─────────────────┤ │
│                        │  │ 📊 课题 1: 行业转化率  ✅完成│ │
│                        │  │ 🔄 课题 2: 竞品定价    调研中│ │
│                        │  │ 💤 课题 3: 用户画像    排队中│ │
│                        │  └────────────────────────────┘ │
│                        ├────────────────────────────────┤
│                        │  输入框                        │ │
└────────────────────────┴────────────────────────────────┘
```

#### 区域 1: 信息流（上方，占 ~70%）

这是 AI 的"发言"区域，按时间线滚动，包含四种卡片：

**① 洞察卡片（💡）**
AI 从讨论中提炼出的新视角、反直觉观点、被忽略的角度。
```
┌─ 💡 洞察 ─────────────────────── 14:32 ─┐
│ 你们在讨论降价，但还没有人提到         │
│ 竞品最近都在涨价。逆势降价可能传递     │
│ "我们不行了"的信号。                    │
└──────────────────────────────────────────┘
```

**② 提问卡片（🤔）**
AI 主动提出的、值得讨论的问题。可以是挑战性的、补充性的、或发散性的。
```
┌─ 🤔 问题 ─────────────────────── 14:35 ─┐
│ 你们的目标用户到底是价格敏感型还是       │
│ 功能驱动型？这个判断会影响后面所有决策。 │
└──────────────────────────────────────────┘
```

**③ 快速核实卡片（🔍）**
对会议中提到的事实性内容的即时查证。
```
┌─ 🔍 快速核实 ──────────────────── 14:37 ─┐
│ "Notion 最近涨价了 20%"                   │
│ → 确认：2025年11月 Notion Plus 从 $8→$10  │
│   涨幅 25%。Team 计划未变。               │
└───────────────────────────────────────────┘
```

**④ 调研结果卡片（📊）**
后台调研完成后推送的详细结果。比其他卡片更长，可折叠。
```
┌─ 📊 调研完成 ──────────────────── 14:41 ─┐
│ 课题: SaaS 行业 2025 年定价趋势           │
│                                           │
│ ▸ 点击展开完整调研结果                     │
│                                           │
│ 关键发现：73% 的 SaaS 公司在过去 12 个月  │
│ 提价，平均涨幅 15%。降价策略主要出现在... │
└───────────────────────────────────────────┘
```

#### 区域 2: 调研队列（下方，占 ~30%，可折叠）

显示正在进行和待处理的调研任务。这个区域让用户知道 AI 在后台忙什么。

每个调研任务是一张小卡片：
```
📊 SaaS 行业定价趋势          ✅ 完成  [查看]
🔄 竞品功能对比分析            🔄 调研中 (45s)
💤 用户价格敏感度调查方法       排队中   [取消]
```

状态流转：`排队中 → 调研中 → 完成`

调研完成后，结果自动推送到上方信息流。

#### 区域 3: 输入框

保持现有功能，但增加两种输入模式：

- **普通提问**：回车发送，AI 基于会议上下文回答
- **下调研任务**：输入 `/research 竞品最近的定价变化` 手动创建调研课题

### AI 主动性的三个层次

关键问题：AI 应该多主动？太安静没价值，太吵会打扰。建议三档可调：

| 层次 | AI 行为 | 适合场景 |
|------|---------|---------|
| **观察者** | 只在用户提问时回应，不主动发言 | 用户想安静开会 |
| **顾问**（默认） | 有重要洞察时主动说，但控制频率（如每 5 分钟最多 1 条） | 日常会议 |
| **研究员** | 积极提问、主动发现课题并调研、高频互动 | 头脑风暴、战略讨论 |

在顶栏用一个小开关切换，或者直接在输入框告诉 AI "安静一点"/"积极一点"。

### AI 调研任务的触发方式

调研课题从两个入口产生：

**1. AI 自动识别并建议**

AI 在分析转写时，识别到值得调研的课题，推送一张建议卡片：

```
┌─ 💡 建议调研 ─────────────────── 14:33 ─┐
│ 会议提到想做海外市场，但还没讨论过       │
│ 目标地区的法规差异。要不要我查一下？     │
│                                          │
│         [去调研]    [跳过]               │
└──────────────────────────────────────────┘
```

用户点"去调研"→ 任务进入调研队列。点"跳过"→ 卡片变灰。
**不点也行**——会议在继续，用户可能没空看。卡片就留在那里，之后再点。

**2. 用户手动指派**

输入框输入自然语言，AI 判断是提问还是调研任务：
- "帮我查一下 Notion 的定价历史" → 调研任务
- "刚才说的那个观点你怎么看" → 即时回答

不需要特殊命令，AI 自己判断。

### 调研的"深度"控制

不是所有调研都需要深度报告。建议分两档：

| 类型 | 耗时 | 输出 | 例子 |
|------|------|------|------|
| **快速核实** | 5-15 秒 | 1-3 句话 | "Apple 抽成比例是多少" |
| **深度调研** | 1-5 分钟 | 结构化报告 | "SaaS 行业 2025 定价趋势" |

AI 自动判断深度。快速核实直接在信息流里回答（🔍 卡片），深度调研进入调研队列。

## 信息密度与注意力管理

会议中用户的注意力是最稀缺的资源。几个设计原则：

### 原则 1: 一眼扫过原则

每张卡片的**第一行**必须能独立传达价值。用户扫一眼就知道要不要细看。

```
❌ "根据我对当前讨论的分析，我认为有一个值得关注的角度..."
✅ "竞品都在涨价，你们却在讨论降价。"
```

### 原则 2: 不打断，只堆叠

AI 的输出永远不弹窗、不弹通知、不抢焦点。只是静静地往信息流里加卡片。用户自己决定什么时候看。

唯一的例外：调研完成时，调研队列的状态变化 + 信息流顶部出现一个**不显眼的小提示**（如数字角标）。

### 原则 3: 收起旧的，突出新的

信息流中超过 10 分钟未互动的卡片自动折叠为一行摘要：

```
── 14:20-14:30 的 3 条洞察（点击展开）──
```

保持信息流始终聚焦在最新内容。

### 原则 4: 让 AI 学会闭嘴

如果转写内容是闲聊、寒暄、与主题无关的对话，AI 应该什么都不说。"没有有价值的洞察"本身就是一个正确的判断。

当前默认 prompt 里已经有 `"如果新增内容没什么值得讨论的，只输出「—」"` 这条规则，是对的。但需要更严格——宁可少说，不要说废话。

## 架构影响

这些交互变化对架构提出了新要求，按影响大小排列：

### 1. 需要联网搜索能力（新增组件）

当前 AI 只能基于转写内容做推理。要做"调研"，必须能搜索互联网。

方案选择：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A: LLM 自带搜索**（如 Perplexity API） | 一次调用返回搜索+总结 | 绑定特定服务商 |
| **B: 搜索 API + LLM 两步**（SerpAPI/Tavily → MiniMax） | 灵活，搜索和总结解耦 | 两次 API 调用，更复杂 |
| **C: MiniMax 插件搜索** | 如果 MiniMax 支持 tool_use/function_calling | 需要确认 API 支持情况 |

建议先调研 MiniMax M2.5 是否支持 function calling / web search 插件。如果不支持，方案 A（Perplexity）最快落地。

### 2. 并发任务管理（新增组件）

当前只有一个串行的 AI 调用。新设计需要：

- 信息流的洞察/提问：持续分析转写，定期输出
- 调研任务：独立的后台任务，可并行多个
- 用户提问：即时响应

需要一个简单的任务队列：

```
ResearchManager
├── 信息流分析（周期性，单线程）
├── 调研队列（FIFO，可并行 1-2 个）
└── 用户提问（最高优先级，插队执行）
```

### 3. AI 输出结构化（Prompt 改造）

当前 AI 输出是自由文本。新设计需要 AI 输出**结构化内容**，以便 App 渲染不同类型的卡片。

两种实现方式：

**方式 A：单次调用，结构化输出**

让 AI 一次性返回 JSON，包含多种类型的输出：

```json
{
  "insights": ["竞品都在涨价..."],
  "questions": ["目标用户是价格敏感型还是功能驱动型？"],
  "fact_checks": [],
  "research_suggestions": [
    {"topic": "SaaS 行业 2025 定价趋势", "reason": "会议讨论了降价策略但缺少行业数据"}
  ]
}
```

**方式 B：多次轻量调用**

把"洞察"、"提问"、"调研建议"拆成独立的 prompt，各自触发。更灵活，但 API 调用次数多。

建议先用方式 A，简单直接。

### 4. 流式响应（复用之前的建议）

信息流里的卡片需要流式填充，调研结果需要边查边显示进度。这一点和之前提的 SSE 流式响应建议一致。

- [ ] 调研能力的搜索 API 选型：Perplexity / Tavily / SerpAPI / 其他？
- [ ] AI 主动性的频率控制怎么调到舒适区？需要实际使用反馈
- [ ] 调研结果的可信度标注：是否需要显示来源链接？
- [ ] 多语言会议场景：AI 洞察和调研用什么语言输出？

---

## 结论

核心改动是两件事：

1. **交互层**：从"聊天式"改为"信息流 + 调研队列"，AI 输出分类为洞察/提问/核实/调研四种卡片
2. **能力层**：给 AI 加上联网搜索能力，让它能真正去调研而不只是基于转写内容推理
